Semantic Versioning

Versioning is an important and one of the rather tricky aspects of software development. Simply put, versioning is assigning a unique number that identifies a specific package or release. In this post, we are going to look into a popular versioning method called Semantic Versioning.

What is Semantic Versioning?

Semantic Versioning, is a popular versioning scheme that mainly uses a three-part version number. The version number is in the following format (there are pre-release versions as well which we will look into in detail later):

For example, if the version number is 2.3.54, the individual parts would mean:

  • 2: Major version
  • 3: Minor version
  • 54: Patch version

The rule of thumb when it comes to incrementing these numbers is:

Major version

Major version number is incremented when you make incompatible changes.

Examples:

You removed an entire public class from a NuGet package

You changed the parameters that API endpoint accepts

Minor version

Minor version is incremented when you add new functionality in a backwards compatible manner.

Examples:

You add a new endpoint to a public API

You add a new parameter to a method with a default value so that old consumers can still call the method without breaking

Patch version

Patch version is incremented when make backwards compatible fixes.

Examples:

You fixed a bug without making backwards incompatible changes

Pre-releases

In addition to the major, minor and patch versions we may want to use pre-release versions. This would indicate that the product is not finalized. Examples of pre-release version:

All the examples above are pre-release versions and they are ordered from lowest precedence to the highest one.

Please note semantic versioning has no knowledge of the words “alpha”, “beta” or “rc”. The comparison is purely done by alphabetical order for non-numeric versions.

Build metadata

In addition to all the release and pre-release versions, we can also use extra metadata by using plus sign (+) as a separator. This part is not used in precedence calculations and has informational purposes only. The metadata that follows the plus sign can be a series of dot-separated identifier list.

Refactoring

How about refactoring? It’s not a major breaking change and you don’t add new functionality. Also it doesn’t count as bugfixes. It might help preventing bugs from being introduced in the future but it doesn’t stricly count as fixing anything.

We can find the answer in the Semantic Versioning specs:

Patch version Z (x.y.Z | x > 0) MUST be incremented if only backwards compatible bug fixes are introduced. A bug fix is defined as an internal change that fixes incorrect behavior.

So if you change the code for whatever reason, you must at least increment the patch version so that you maintain uniqueness of the package/release.

Front-end Versioning

A common question and debated issue is how to version front-ends. The Semantic Versioning specs is all about “API changes”. API in this context can refer to a HTTP API or a package (NuGet, npm, Maven etc).

Front-ends are consumed by end-users and they are not consumed by other software.

By saying not consumed by other software, I’m not counting the webscrapers. When you develop a front-end page you don’t make a contract with an external tool whose goal is to scrape data from your page. Most likely, it happens without your consent. Therefore, if you make a change in your markup that breaks webscrapers it doesn’t constitute breaking change in an API

Defining a “breaking change” in a front-end is not easy. If you move a functionality to another page, it might be seen as breaking change as some users might fail to find the new location of the functionality. Does this mean that you should increment the major version? If you did, what would that mean to the end-users? They still need to use the application/website just like they did before. I think when interacting with actual human users, version numbers don’t mean much.

Users don’t care about your versions

The only exception I can think of is a complete overhaul of the project. If everything changes so drastically, you may choose to refer to it as version 2.0 of your application.

Other than that, I’d argue the best way to handle changes in a user-facing application is leveraging changelogs.

When a user logs in, you can display them a nice little pop-up and briefly explain the key things that changed. Some more complicated changes might need some interactive walkthroughs etc. The main point is, a user doesn’t want to or need to know that you deployed 2.10.24 version of your application. Just show them how it effects their experience and leave the technical details out.

Conclusion

Semantic Versioning is a very popular versioning scheme as it’s simple and quite flexible. There are some aspects that are debatable whether or not it should be used such as front-end versioning but this doesn’t mean that it does a good job most of the time. We also discussed how to handle versioning in user-facing applications.

Solana Epoch Monitor

Solana is currently one of the popular blockchains. Every few days a new epoch starts at which point the staking rewards are sent to the stakers. This post’s main focus is not cryptocurrency, rather it’s about a tool that monitors Solana blockchain and sends email notifications when a new epoch starts. So I’m not going to deep into Solana terminology but just to understand the basics, here are a few definitions:

  • Solana: A Blockchain technology designed to host decentralized and scalable applications.
  • Staking: The process of locking up cryptocurrency in order to earn interest
  • Epoch: An epoch is the length of a certain amount of blocks. This is a variable duration and typically it’s somewhere between 2-4 days.

If you are an Solana investor staking your Solana tokens, then at the end of the epoch you earn some interest. This is where this notification tool comes in. If your wallet doesn’t have a feature of notifying you when you receive staking rewards, you can use this tool to receive email notifications when an epoch ends. This way you can check your wallet and keep an eye on your rewards.

Implementation

If you would like to check the current epoch manually, you can always go to Solana explorer page which shows the current epoch.
